[Normal] Optimus Mexico Behind-the-Meter Investor Fixes
[Normal] Developer-ready design, flow and implementation plan
[Normal] Module target: Optimus_Mexico_Behind_TheMeter | Fix package: BTM investment-grade workflow
[Normal] Document objective: translate the consultant/investor review into programmable changes. The fixes make the app safe for small industrial investor screening and more credible for bank/lender review. They do not replace legal, tax, credit or engineering diligence.
[Heading 1] 1. Executive Summary
[Normal] The current Optimus Mexico app already includes front-of-meter node/PML/PPA/capacity tools and a Behind-the-Meter CFE module. The three critical fixes are designed to prevent users from confusing merchant revenues with behind-the-meter savings, to improve dispatch accuracy through LP optimization, and to prevent investment recommendations when the input data is synthetic or insufficient.
[Heading 1] 2. Current App Context
[Normal] The README indicates the current app has Node Resolver, PML Scraper, PV Loader, Dispatch, Optimizer, PPA/CFE mixed-development, Capacity, Printout and Behind-the-Meter CFE modules. The BTM module already includes tariff calendar, CFE bill reconstruction, rule-based dispatch, investor cases, red flags and GO/REVISE/NO-GO logic. The next step is to strengthen these with a strict BTM-only investor workflow, LP dispatch and data-quality gates.
[Heading 1] 3. Target User Journeys After Fixes
[Heading 1] 4. Fix 1 - Separate Bankable BTM Investor Case from Merchant/PPA Logic
[Heading 2] 4.1 Design intent
[Normal] When project_mode = BTM_CFE, the app must disable merchant/PML/PPA/CENACE capacity revenue streams. Behind-the-meter investment should be based on avoided CFE bill components, PV self-consumption and optional backup/resilience value only.

ALLOWED_BTM_VALUE_STREAMS = [
    "cfe_energy_savings",
    "cfe_capacity_savings",
    "cfe_distribution_savings",
    "pv_self_consumption_value",
    "backup_value_optional"
]

DISABLED_IN_BTM = [
    "pml_merchant_revenue",
    "ppa_revenue",
    "cenace_capacity_revenue",
    "ancillary_services_revenue"
]

def enforce_btm_revenue_scope(project_mode: str, selected_streams: list[str]) -> dict:
    disabled = []
    allowed = selected_streams
    if project_mode == "BTM_CFE":
        disabled = [s for s in selected_streams if s in DISABLED_IN_BTM]
        allowed = [s for s in selected_streams if s in ALLOWED_BTM_VALUE_STREAMS]
    return {
        "allowed_streams": allowed,
        "disabled_streams": disabled,
        "is_valid": len(disabled) == 0,
        "message": "BTM_CFE mode excludes merchant/PPA/CENACE capacity revenues." if disabled else "OK"
    }

[Heading 1] 5. Fix 2 - Add LP Optimizer as Bankable BTM Dispatch Engine
[Heading 2] 5.1 Design intent
[Normal] Rule-based dispatch remains as a fast screening engine. The new LP optimizer becomes the default for Bankable Investor Case. It must jointly optimize demand charge reduction, energy period shifting, PV self-consumption, no-export compliance, degradation cost and optional reserve SOC.
[Heading 2] 5.2 New file
[CodeBlock] agents/btm_lp_optimizer_agent.py
[Heading 2] 5.3 LP objective
[CodeBlock] Minimize:
  total_energy_charge_mxn
+ capacity_charge_mxn
+ distribution_charge_mxn
+ degradation_cost_mxn
+ optional_backup_reserve_penalty
+ optional_unserved_energy_penalty
[Heading 2] 5.4 Decision variables
[Heading 2] 5.5 Constraints
[CodeBlock] For each 15-minute interval t:
0 <= charge_pv_kw[t] <= bess_power_kw
0 <= charge_grid_kw[t] <= bess_power_kw if grid_charge_enabled else 0
0 <= discharge_kw[t] <= bess_power_kw
soc_min_kwh <= soc_kwh[t] <= soc_max_kwh
grid_import_kw[t] >= 0
grid_export_kw[t] == 0 if no_export_mode else >= 0

SOC balance:
soc[t] = soc[t-1] + 0.25 * eta_charge * (charge_pv_kw[t] + charge_grid_kw[t]) - 0.25 * discharge_kw[t] / eta_discharge

Load balance:
load_kw[t] = pv_to_load_kw[t] + discharge_kw[t] + grid_import_kw[t]

PV balance:
pv_kw[t] = pv_to_load_kw[t] + charge_pv_kw[t] + pv_curtailed_kw[t] + grid_export_kw[t]

Demand constraints by month m:
Dmaxpunta_after[m] >= grid_import_kw[t] for all t in punta period in month m
Dmaxmensual_after[m] >= grid_import_kw[t] for all t in month m
billing_capacity_kw[m] = min_proxy(Dmaxpunta_after[m], Qmensual_after / (24 * days * factor_carga))
billing_distribution_kw[m] = min_proxy(Dmaxmensual_after[m], Qmensual_after / (24 * days * factor_carga))
[Heading 2] 5.6 Handling min() in LP
[Normal] Because min() is non-linear, implement the billing-demand calculation after optimization for v1, or use auxiliary variables and conservative formulation in v2. For MVP bankable LP, optimize against measured peak reduction and energy cost, then calculate final bill with the existing CFE bill engine. Use an iterative target-cap loop if exact demand-charge optimization is required without mixed-integer variables.

[Heading 2] 6.6 Pseudocode
[CodeBlock] def evaluate_investment_readiness(customer_files, meter_quality, bill_reconstruction, project_config):
    issues = []
    warnings = []

    if project_config.get("uses_synthetic_data", False):
        return status("DEMO", ["Synthetic sample data in use"], allowed=["view_demo", "export_demo_report"])

    if customer_files.get("cfe_bill_months", 0) < 12:
        issues.append("Upload 12 monthly CFE bills")
    if meter_quality.get("months_15min", 0) < 12:
        issues.append("Upload 12 months of 15-minute load data")
    if meter_quality.get("missing_interval_pct", 100) > 1:
        issues.append("15-minute meter data has more than 1% missing intervals")
    if not project_config.get("tariff_confirmed", False):
        issues.append("Confirm CFE tariff and division")

    if issues:
        return status("SCREENING", issues, allowed=["indicative_savings", "data_request"])

    if bill_reconstruction.get("annual_error_pct", 100) > 3:
        return status("REVISE", ["Baseline bill reconstruction error exceeds 3%"], allowed=["qa_report"])

    if bill_reconstruction.get("months_failed", 0) > 2:
        return status("REVISE", ["Too many monthly reconstruction failures"], allowed=["qa_report"])

    return status("INVESTMENT_READY", warnings, allowed=["go_revise_nogo", "investor_report", "evidence_package"])

[Heading 1] 7. End-to-End Flow After Fixes
[CodeBlock] 1. User enters BTM module
2. App sets project_mode = BTM_CFE and activates revenue guard
3. User uploads CFE bills, 15-min load, tariff data, PV profile if needed
4. Data quality engine validates timestamps, missing intervals, tariff coverage and synthetic flags
5. Baseline bill engine reconstructs CFE bills and calculates error
6. Investment readiness agent labels case: DEMO, SCREENING, REVISE or INVESTMENT READY
7. If SCREENING/REVISE: show indicative output and data correction actions only
8. If INVESTMENT READY: user selects BESS/PV+BESS sizing scenarios
9. LP optimizer runs bankable dispatch; rule-based engine can be shown for comparison
10. Bill engine calculates optimized CFE bill using the same tariff logic as baseline
11. Investor agent applies haircuts, financing terms and red-flag logic
12. App produces customer dashboard, investor case and evidence package
